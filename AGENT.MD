# Telezhka Finance Bot: память проекта

Этот файл нужен как подробная память проекта. Его должен суметь прочитать владелец проекта, другой инженер, другая нейронка или будущий Codex, чтобы быстро восстановить контекст, историю решений, текущую архитектуру и незакрытые проблемы.

Важно: файл специально не содержит реальные секреты: Telegram-токен, пароль сервера, Google service account JSON, приватные SSH-ключи, секрет Эвотора. Они уже встречались в чате/на скриншотах, но хранить их в репозитории нельзя. Если есть риск утечки, их нужно ротировать.

## 1. Что строим

Проект: production-ready Telegram-бот для финансового учета небольшой команды через Google Sheets и серверную базу.

Главная задача:

- сотрудники/владелец пишут выручку свободным текстом в Telegram;
- бот понимает имя, дату, наличные, безналичные;
- бот записывает данные в Google Sheets;
- бот считает зарплаты, выручку, чистую прибыль;
- бот присылает отчеты и напоминания;
- бот должен быть стабильным 24/7 на VPS;
- позже подключается Эвотор, чтобы выручка подтягивалась автоматически по чекам.

Изначальные примеры сообщений:

```text
Ксюша
нал 12500
безнал 38640
```

```text
2 июля Настя нал 14000 безнал 42000
```

Позже добавленные реальные сценарии:

```text
ксюша+дима нал 500 безнал 1000
сегодня нал 13000 безнал 28000
выручка за пятницу
выручка за 03.07
измени наличку за 2 июля на 19000
сегодня не работаем
```

## 2. Текущий стек

Выбранный стек:

- Python;
- aiogram 3 для Telegram-бота;
- SQLAlchemy async;
- PostgreSQL в Docker на сервере;
- SQLite как локальный/dev fallback;
- Alembic для миграций;
- Google Sheets через service account;
- Docker Compose для запуска;
- aiohttp для HTTP-приемника Эвотора;
- JSON-логирование.

Причина выбора:

- Python удобен для Telegram-бота, парсинга текста и интеграций;
- PostgreSQL надежнее SQLite для продакшена;
- Google Sheets остается основной рабочей таблицей для владельца бизнеса;
- Docker Compose упрощает перенос на сервер;
- Google Sheets + PostgreSQL дают и удобный UI, и историю/надежность.

## 4. Бизнес-правила зарплаты

Низкая ставка:

- Ксюша: 2000;
- Настя: 2000;
- Кристина: 2000.

Все остальные:

- 2500.

В `.env` это задается так:

```text
DEFAULT_SALARY=2500
LOW_SALARY=2000
LOW_SALARY_EMPLOYEES=Ксюша,Настя,Кристина
```

Важные правила:

- если продавец один, зарплата считается по его ставке;
- если продавцов несколько, например `КСЮША+ДИМА`, зарплата суммируется;
- `&` в таблице и отчетах означает "все остальные", кроме Ксюши, Насти и Кристины;
- если день нерабочий, зарплата должна быть 0, даже если продавец уже указан в расписании.

Примеры:

```text
Ксюша -> 2000
Дима -> 2500
Ксюша+Дима -> 4500
Настя+Ксюша -> 4000
Кристина+Настя -> 4000
Ксюша+& -> 4500
```

## 6. Google Sheets

Основная таблица:

```text
ТЕЛЕЖКА - 2026
```

Spreadsheet ID:

```text
1tZ1JwIrK0co4l3LRKSn8wODdC6XuXmXamlJ2-Tjc_FE
```

Существующие листы:

- МАЙ 2026;
- ИЮНЬ 2026;
- ИЮЛЬ 2026;
- АВГУСТ 2026;
- СЕНТЯБРЬ 2026.

Основные колонки:

- A: дата;
- B: продавец;
- C: зарплата;
- D: наличный расчет;
- E: безналичный расчет.

Блоки недельной зарплаты:

- G: имена/категории;
- H: зарплата за неделю по человеку/категории;
- в G уже стоят Ксюша, Настя, Кристина, `&`.

Важно:

- бот не должен создавать хаос в структуре таблицы;
- бот ищет строку по дате;
- бот выбирает лист по месяцу даты;
- если дата в июле, запись идет на лист `ИЮЛЬ 2026`;
- если дата в июне, запись идет на лист `ИЮНЬ 2026`;
- при переходе месяца бот должен автоматически писать в правильный лист.

## 7. Правила отчетов

Недельный отчет:

- неделя считается с понедельника по воскресенье;
- если неделя пересекает границу месяца, недельный отчет обязан брать данные с двух листов;
- пример: понедельник и вторник в июне, среда-воскресенье в июле -> отчет недели должен включить и июнь, и июль;
- это касается и недельных зарплат в колонке H.

Месячный отчет:

- месяц считается строго с 1 числа текущего месяца по выбранный/сегодняшний день;
- месячный отчет НЕ должен подтягивать понедельник/вторник из прошлого месяца, даже если они входят в календарную неделю;
- когда наступит 1 августа, месячный отчет должен стать августовским: 01.08 - текущий день августа;
- июльский месячный отчет останется июльским только если явно смотреть июль или данные в таблице.

Статистика должна считать:

- общий нал;
- общий безнал;
- общую выручку;
- сумму зарплат;
- чистую прибыль;
- количество записей.

## 8. Парсер свободного текста

Парсер должен быть терпимым к формулировкам.

Поддерживаемые варианты налички:

```text
нал
наличка
наличку
наличные
кэш
cash
```

Поддерживаемые варианты безнала:

```text
безнал
безналичные
картой
карта
терминал
эквайринг
card
```

Даты:

- если дата не указана, используется сегодняшняя дата по `Europe/Moscow`;
- `2 июля`;
- `03.07`;
- `за пятницу`;
- `за среду`;
- `сегодня`;
- `вчера`;
- допущена опечатка `сегорлня`/похожие варианты.

Если пользователь пишет:

```text
сегодня нал 13000 безнал 28000
```

и сотрудник не указан, бот должен взять сотрудника из таблицы, из колонки B по этой дате. Это было добавлено после реального кейса, когда Эвотор выгрузил нули, а владелец хотел вручную поправить выручку без повторного ввода сотрудника.

## 9. Расписание

Добавлена возможность отправить расписание пачкой.

Пример:

```text
пн. 29.06 — Ксюша
вт. 30.06 — Настя

ср. 01.07 — Настя
чт. 02.07 — Дима
пт. 03.07 — Ксюша + Дима
сб. 04.07 — Настя + Ксюша
вс. 05.07 — Кристина + Ксюша

пн. 06.07 — Кристина
вт. 07.07 — Настя
ср. 08.07 — Ксюша
чт. 09.07 — Настя
пт. 10.07 — Ксюша + Кристина
сб. 11.07 — Кристина + Настя
вс. 12.07 — Ксюша + &
```

Что бот делает:

- понимает строки расписания;
- пишет продавца в колонку B соответствующего дня;
- сохраняет `КСЮША+ДИМА` как составное имя, не отрезает второго сотрудника;
- `?` или неопределенный сотрудник может быть превращен в `&`, если это нужно для категории "остальные".

Однажды была проблема: бот писал расписание в таблицу, но отвечал "не смог записать". Это было исправлено: расписание должно возвращать успешный ответ.

## 10. Исправления записей

Бот должен уметь изменять уже внесенный день.

Пример:

```text
измени наличку за 2 июля на 19000
```

Ожидание:

- бот находит строку за 2 июля;
- меняет только наличку;
- сохраняет остальные данные;
- пересчитывает выручку и зарплату при необходимости.

Важный кейс:

- если на 3 число изменили продавца с `КСЮША+ДИМА` на `ДИМА`, зарплата должна пересчитаться;
- если сотрудник меняется, недельная зарплата в H тоже должна обновиться.

## 11. Нерабочие дни

Фраза:

```text
сегодня не работаем
```

означает:

- день записывается как рабочая строка с нулевой выручкой;
- наличка = 0;
- безнал = 0;
- зарплата = 0;
- продавец берется из заранее внесенного расписания, если он есть в колонке B;
- если продавца нет, бот должен попросить уточнить или не падать.

Важное исправление:

- раньше бот мог сохранить сотрудника из расписания, но оставить зарплату;
- это неправильно;
- сейчас правило: если не работаем, зарплата всегда 0.

## 12. Напоминания и автоматические сообщения

Изначально нужно было каждый день в 22:30 напоминать внести выручку.

Позже после Эвотора логика поменялась:

- ежедневное время стало 22:10;
- если включен Эвотор, бот в 22:10 должен сам собрать выручку из Эвотора, записать в таблицу и прислать сводку за сегодня;
- если Эвотор выключен или не настроен, бот может работать как обычная напоминалка;
- если за сегодняшний день запись уже есть, лишнее напоминание не нужно.

Также добавлено:

- каждое воскресенье в 23:00 бот присылает недельный отчет;
- каждое 1 число месяца в 10:00 бот напоминает об оплате сервера.

Переменные:

```text

WEEKLY_REPORT_TIME=23:00
WEEKLY_REPORT_WEEKDAY=6

SERVER_PAYMENT_REMINDER_DAY=1
```

## 14. Docker

Основной запуск:

```text
docker compose up -d --build
```

Проверка:

```text
docker compose ps
docker compose logs --tail=100 bot
```

В `compose.yaml`:

- bot зависит от postgres;
- postgres хранит данные в volume `postgres_data`;
- локальная папка `./data` монтируется в контейнер как `/app/data`;
- Google JSON монтируется read-only;
- порт бота проброшен как `8080:8080`;
- Telegram API зафиксирован через `extra_hosts`, чтобы обходить возможные DNS/сетевые проблемы:

```text
api.telegram.org:149.154.167.220
```

Было важно не пробрасывать порт 80 напрямую на контейнер, потому что порт 80 занят nginx. Правильная схема:

```text
internet -> nginx:80 -> bot:8080
```

## 16. SSH: текущая важная нерешенная проблема

Проблема:

- вход по SSH с Mac на сервер нестабилен или не работает;
- веб/VNC-консоль провайдера работает;
- ключевой вход планировалось оставить;
- вход по паролю хотели отключить;
- но нормальный удобный SSH с Mac пока не доведен до идеального состояния.

Важно: проблема похожа не на неверный ключ, а на сетевой/sshd/banner/firewall/provider issue. В прошлых попытках уже были похожие симптомы на другом сервере.

Пользователь предпочитает инструкции через `nano`, потому что вставка в VNC-консоль часто ломается. Если нужно менять конфиги через VNC, давать команды так, чтобы можно было открыть файл в `nano` и руками вставить короткий блок.

Ключ на Mac:

```text
~/.ssh/telezhka_finance_bot
```

Приватный ключ нельзя публиковать или коммитить.

Команды проверки на сервере:

```text
systemctl status ssh --no-pager
ss -tlnp
sshd -t
journalctl -u ssh --no-pager -n 80
```

Рекомендуемый SSH-конфиг через `nano`:

```text
nano /etc/ssh/sshd_config.d/20-telezhka.conf
```

Содержимое:

```text
UseDNS no
GSSAPIAuthentication no
LoginGraceTime 30
MaxStartups 100:30:200

PubkeyAuthentication yes
PasswordAuthentication no
```

Проверка и перезапуск:

```text

systemctl status ssh --no-pager
```

```text

```

Содержимое:

```text

```

Потом:

```text

```

```text

```

До исправления SSH можно продолжать через VNC-консоль, но это неудобно и рискованно для долгой эксплуатации.

## 17. nginx и публичный HTTP

Для Эвотора нужен публичный HTTP endpoint.

Правильная схема:

```text
Эвотор -> http://91.107.122.0/evotor/... -> nginx -> http://127.0.0.1:8080/evotor/...
```

Файл nginx:

```text
/etc/nginx/conf.d/telezhka.conf
```

Рекомендуемое содержимое:

```nginx
server {
    listen 80 default_server;
    listen [::]:80 default_server;
    server_name _ 91.107.122.0;

    location /health {
        proxy_pass http://127.0.0.1:8080/health;
        proxy_http_version 1.1;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_connect_timeout 5s;
        proxy_read_timeout 30s;
    }

    location /evotor/ {
        proxy_pass http://127.0.0.1:8080/evotor/;
        proxy_http_version 1.1;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_connect_timeout 5s;
        proxy_read_timeout 30s;
    }
}
```

Проверка:

```text
nginx -t
systemctl restart nginx
curl -i -H "Host: 91.107.122.0" http://127.0.0.1/health
```

Ожидаемый ответ:

```text
HTTP/1.1 200 OK
{"status":"ok"}
```

Была ошибка:

```text
unexpected end of file, expecting ";" or "}"
```

## 20. Эвотор: текущая проблема

Симптом:

- в личном кабинете Эвотора выручка видна;
- например была выручка 120 рублей, позже 10010 рублей;
- команда `эвотор` в Telegram возвращает нули;
- значит бот не видит чеки.

Возможные причины:

1. Эвотор еще не отправляет webhooks, потому что приложение в тестовом режиме не привязано к боевому аккаунту/кассе.
2. URL недоступен снаружи.
3. nginx не проксирует `/evotor/...` на контейнер.
4. порт 80 занят nginx, а контейнер раньше пытался занять 80 напрямую.
5. в `.env` была опечатка `EVOTOR_RECEIPTS_ENABLES`, должно быть `EVOTOR_RECEIPTS_ENABLED`.
6. Эвотор отправляет не на тот endpoint.
7. Эвотор отправляет payload с другой структурой, и парсер пока его не узнает.
8. Нужны дополнительные поля в разделе "Чеки (ver.2)".
9. Секрет авторизации не совпадает.

Важно: текущий режим receipt-file не читает выручку из Dashboard Эвотора напрямую. Он считает только те чеки, которые Эвотор реально прислал на наш сервер и которые сохранились в:

```text
./data/evotor-receipts.jsonl
```

Если файл пустой или `receipts_count=0`, команда `эвотор` будет показывать нули.

## 22. Эвотор: ручной тест приема чека

Этот тест нужен, чтобы отделить проблему "бот не умеет считать" от проблемы "Эвотор ничего не присылает".

Вставить вместо `<EVOTOR_CALLBACK_SECRET>` реальный секрет из `.env`.

```text
curl -i -X POST http://127.0.0.1:8080/evotor/receipts \
  -H "Content-Type: application/json" \
  -H "X-Authorization: <EVOTOR_CALLBACK_SECRET>" \
  -d '{"type":"SELL","deviceId":"00307900861869","dateTime":"2026-07-05T12:00:00+03:00","totalAmount":120,"paymentSource":"CASH"}'
```

После этого:

```text
curl -i http://127.0.0.1:8080/evotor/status
```

Если `receipts_count` вырос, приемник работает.

Дальше в Telegram команда:

```text
эвотор
```

должна увидеть этот тестовый чек, если дата совпадает с сегодняшней датой в Москве.

Если тестовый чек работает, а реальные чеки нет, проблема на стороне настроек Эвотора/статуса приложения/полей/URL.

Если тестовый чек не работает, проблема в нашем приемнике, `.env`, nginx или Docker.

## 23. Как бот считает Эвотор

Файл:

```text
app/integrations/evotor.py
```

Есть два режима:

1. API pull через `EVOTOR_REVENUE_URL_TEMPLATE`.
2. Receipt-file mode через `EVOTOR_RECEIPTS_FILE`.

Сейчас основной рабочий режим - receipt-file mode:

- Эвотор шлет чек на `/evotor/receipts`;
- приемник сохраняет payload в `data/evotor-receipts.jsonl`;
- команда `эвотор` читает этот файл;
- бот фильтрует чеки по дате;
- бот фильтрует по терминалу `00307900861869`;
- бот суммирует наличку/безнал;
- возвраты вычитает.

Поля терминала, которые бот ищет:

- `terminalUuid`;
- `terminal_uuid`;
- `terminalId`;
- `terminal_id`;
- `deviceUuid`;
- `device_uuid`;
- `deviceId`;
- `device_id`;
- `cashRegisterId`;
- `cash_register_id`;
- `kkm`.

Если поле не найдено, бот также ищет ID терминала в JSON-тексте.

Поля оплат:

- cash/cashless markers;
- `paymentSource`;
- `payments`;
- `paymentType`;
- похожие поля.

Возвраты:

- `payback`;
- `return`;
- `refund`;
- `возврат`.

Если структура реального payload отличается, нужно посмотреть `last_receipt_keys` в `/evotor/status` и добавить поддержку в `app/integrations/evotor.py`.

### Продолжение (та же сессия): план на stunnel (SSH через 443/TLS)

Поскольку смена порта на 2222 не решила проблему, рабочая гипотеза сузилась до "режется/теряется именно незашифрованная сигнатура SSH-баннера" (DPI-подобное поведение), а не порт как таковой. Решили попробовать замаскировать SSH под TLS-трафик на 443 порту через `stunnel4`: сервер принимает TLS-соединение на 443 и проксирует расшифрованный поток на локальный `127.0.0.1:22`, SSH-протокол внутри остается тем же (аутентификация и шифрование самого SSH не меняются, TLS — только маскировка транспорта).

Предложенный план (ЕЩЕ НЕ ВЫПОЛНЕН, ждем результат):
1. `ss -tlnp | grep 443` — убедиться, что порт свободен (nginx слушает только 80).
2. `apt install -y stunnel4`.
3. Сгенерировать самоподписанный сертификат:
   ```
   openssl req -new -x509 -days 3650 -nodes -out /etc/stunnel/stunnel.pem -keyout /etc/stunnel/stunnel.pem -subj "/CN=91.107.122.0"
   chmod 600 /etc/stunnel/stunnel.pem
   ```
4. Конфиг `/etc/stunnel/stunnel.conf`:
   ```
   pid = /var/run/stunnel4.pid
   [ssh]
   accept = 443
   connect = 127.0.0.1:22
   cert = /etc/stunnel/stunnel.pem
   ```
5. В `/etc/default/stunnel4` выставить `ENABLED=1`, затем `systemctl restart stunnel4 && systemctl enable stunnel4`, проверить `ss -tlnp | grep 443`.

Дальше (после подтверждения, что stunnel поднялся на 443) — предстоит настроить клиент на Mac (локальный stunnel/socat туннель на 127.0.0.1, и SSH уже через него). Этот шаг пока не описан, будет зафиксирован после первого успешного теста.

### Сессия 10.07.2026: дополнительная SSH-диагностика с Mac

После коммита попытались выкатить изменения на сервер командой через SSH (`cd /opt/telezhka-finance-bot && git pull && docker compose up -d --build`). Подключение не дошло до выполнения команд:

```
Connection timed out during banner exchange
Connection to 91.107.122.0 port 22 timed out
```

Новые проверки с Mac:

1. `route get 91.107.122.0` показал, что трафик до сервера идет через интерфейс `utun11`. Это VPN-туннель, а не прямой маршрут через Wi-Fi.
2. `netstat -rn -f inet` подтвердил приоритетный default-route через `utun11`; обычный Wi-Fi gateway `192.168.0.1` есть, но не является первым маршрутом для внешнего интернета.
3. `nc -vz -w 8 91.107.122.0 22` показал, что TCP-порт 22 формально открывается: `Connection ... succeeded`.
4. `ssh -vvv -o IPQoS=none ... root@91.107.122.0 true` не помог: TCP-соединение устанавливается, локальный SSH-клиент отправляет `SSH-2.0-OpenSSH_9.8`, затем снова ждет баннер сервера и падает по timeout.
5. `nc -w 8 91.107.122.0 22` не вывел баннер сервера. Это совпадает с прошлой tcpdump-картиной: сервер баннер отправляет, но до клиента он не доходит или ACK обратно не возвращается.
6. Попытка HTTP-проверки `http://91.107.122.0/health` с Mac также зависла до ручного прерывания. На сервере локальный health ранее работал через `127.0.0.1:8080/health`, поэтому проблема похожа не на приложение, а на внешний сетевой путь.
7. Проверка `IPQoS=none` сняла одну возможную гипотезу: дело не в DSCP/TOS-флагах SSH-клиента.

Текущий вывод:

1. Серверный `sshd` ранее уже доказан рабочим: он слушает порт 22, проходит `sshd -t`, отправляет SSH-баннер по tcpdump.
2. Порт 22 с Mac открывается, значит это не обычный `connection refused` и не закрытый порт.
3. Срыв происходит после TCP connect, на доставке первого полезного payload-пакета SSH.
4. Так как текущий маршрут идет через `utun11`, наиболее вероятные причины: VPN, его split/full-tunnel маршрутизация, фильтрация у VPN-провайдера, либо связка VPN + маршрут до сети FirstVDS.
5. Домашний роутер/интернет тоже возможны, но прямо сейчас VPN выглядит более вероятным фактором, потому что маршрут до сервера явно идет через VPN.

Следующие разумные шаги:

1. Проверить тот же SSH без VPN. Если сразу заработает — причина почти точно VPN/его маршрут.
2. Проверить с другой сети (например, мобильный интернет без VPN). Это отделит проблему домашнего роутера/провайдера от VPN.
3. Если без VPN отключаться неудобно или проблема сохранится, реализовать обход через TLS/443 (`stunnel`) или другой публичный endpoint, чтобы SSH-трафик не выглядел как обычный SSH-баннер.
4. До восстановления SSH деплой на сервер возможен через VNC/панель: зайти в `/opt/telezhka-finance-bot`, выполнить `git pull`, затем `docker compose up -d --build`.
